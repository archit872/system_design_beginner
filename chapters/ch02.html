CHAPTER 1

Orientation — What is System Design?

System design is the art of turning ambiguous requirements into a robust architecture. You’ll learn why it matters, how interviews differ from day-to-day engineering, and how to reason with core goals: scalability, reliability, and maintainability. We also establish key terms such as API, latency, and throughput—the language you’ll use for the rest of the book.

Prerequisites

Basic programming knowledge (variables, HTTP requests, simple data structures). If you can build a minimal REST endpoint and consume it from a script, you’re ready.

Learning Objectives

  • Explain why system design matters for product outcomes and team velocity.
  • Differentiate coding interviews from system design interviews.
  • Describe the core goals: scalability, reliability, maintainability.
  • Use foundational vocabulary (API, latency, throughput, SLO, SLA, availability).

1.1 What System Design Means in Industry

In industry, system design is a workflow: clarify requirements, identify constraints, propose architecture, evaluate trade-offs, and iterate. The outcome isn’t just diagrams; it’s a shared plan that coordinates teams and informs future change. Good design reduces rework and accelerates delivery because developers can make local decisions without constantly re-negotiating global ones.

Think of design like city planning. Streets (APIs) connect neighborhoods (services); speed limits (SLOs) balance safety and throughput; utilities (databases, queues, caches) ensure reliable daily life. If you widen roads for more throughput, you might increase noise (cost/complexity). This analogy helps you remember that design is the management of constraints under real-world limits.

Analogy: A code module is a room; a system is a building. You can furnish a room any way you like, but the building must obey weight limits, fire exits, and shared utilities.

Mini case study: A feature launch under load

A small social app is preparing a “stories” feature. Marketing expects a 5× spike on launch day. The team has one week. They consider two paths:

  • Path A: Ship immediately using current monolith; rely on vertical scaling (bigger VM) and a CDN to cache media.
  • Path B: Extract a “stories-service,” add a load balancer, and integrate a queue for uploads with background processing.

Path A is fast but risks downtime if the database becomes a bottleneck. Path B improves reliability and scalability but may slip the deadline. The team chooses A plus a simple rate limit and an emergency read replica. They also define clear SLOs for media latency. The feature launches with short slowdowns but no outage—an acceptable trade-off given the time constraint.

Trade-off highlight: Deliver now with mitigations vs. re-architect for the ideal end state. Optimize for the current constraint you can’t change (time) while keeping a path to evolve.

1.2 System Design in Interviews vs. Practice

Interviews compress months of learning into 45–60 minutes. You demonstrate structured thinking under time pressure: clarify requirements, pick a sensible baseline, call out risks, and communicate trade-offs. Real projects add long-running factors like compliance, org constraints, staged rollouts, on-call experience, and cost models.

Compare & Contrast

Dimension Interview Practice
Scope Single session; simplified constraints Evolving scope; shifting requirements
Metrics Back-of-the-envelope estimations Measured SLIs, budgets, SLAs
Risk Hypothetical failures Real incidents; postmortems
Artifacts Whiteboard diagrams, spoken trade-offs Design docs, runbooks, dashboards
Common pitfall: jumping into micro-optimizations (e.g., sharding) before stating baseline requirements. Always confirm what you’re solving before how.

1.3 Core Goals: Scalability, Reliability, Maintainability

These goals guide most decisions. You rarely maximize all three—design is about balancing them for your product’s stage and constraints.

Scalability

Scalability is a system’s ability to handle increased load without unacceptable latency or cost. Horizontal scaling (more instances) is usually preferred over vertical (bigger instance) beyond early stages. Watch state: the more state a service holds, the harder it is to scale.

Reliability

Reliability means doing what you promise, consistently. It is often measured via availability (e.g., 99.9%) and quantified by SLOs and SLIs. Techniques include redundancy, idempotency, and failover. Reliability also accounts for tail latency, not just averages.

Maintainability

Maintainability is the ease of changing and operating a system. Clear boundaries, good observability, and automation (tests, CI/CD) pay dividends. A highly scalable system that only two people understand is a long-term liability.

Trade-off: Building for 10× the current scale may reduce maintainability now. Prefer an evolutionary path: baseline → measure → refactor with data.

1.4 Key Vocabulary

Learn and use these consistently:

  • API — Boundary for requests/responses between components.
  • Latency — Time to serve a request (often p50/p95/p99).
  • Throughput — Requests or data processed per unit time.
  • Capacity planning — Estimating resources to meet demand.
  • Bottleneck — The slowest component that limits system performance.
  • Rate limiting — Controlling request rate to protect services.
  • Queue — Buffer that decouples producers/consumers.
  • Availability — Proportion of time a service is usable.
  • SLO / SLI / SLA — Objective/indicator/agreement around reliability.
  • Tail latency — Slowest responses that dominate user pain.
Misuse warning: Don’t conflate latency with throughput. A system can have high throughput but still feel slow to users due to high tail latency.

Concrete Examples

Example A: API latency vs. user-perceived speed

A feed API responds in 120 ms (p50) but image loads block rendering for 1.5 s. The fix is not optimizing the API first; it’s adding image placeholders, compressing assets, and using a CDN. System design sees the whole request path.

Example B: Throughput gain by queueing

Video uploads overwhelm a single server doing sync processing. Introduce a queue and background workers; the upload API returns quickly while processing continues. Median latency improves; throughput increases by parallelism.

Practice

  1. Daily systems audit (20–30 min): List three systems you use (WhatsApp, YouTube, Maps). For each, write 3–5 key features and at least one performance challenge (e.g., end-to-end encryption vs. group size). Success: Three lists plus one hypothesized bottleneck each.
  2. Glossary drill (15 min): Write your own one-sentence definition for latency, throughput, availability. Success: Definitions use measurable phrasing (e.g., “p95 under 300 ms”).
  3. Request path sketch (25–35 min): Draw the request path from a browser to a simple API and back (DNS → CDN → LB → app → DB). Success: Each hop labeled with potential failure modes.

Mastery Check

  1. Define: What is latency? What’s the difference between p50 and p95?
    Sample answer

    Latency is the time from request to response. p50 is median (half are faster), p95 highlights tail; improving p95 often improves user experience more than p50.

  2. Explain: Why is throughput not the same as latency?
    Sample answer

    Throughput measures volume/time; latency measures time/request. You can increase throughput via parallelism while leaving latency unchanged or even worse for individual requests.

  3. Apply: A read-heavy API has 400 ms p95 due to DB load. Name two changes.
    Sample answer

    Add a cache layer (CDN or Redis) for hot reads; add a read replica and route reads; denormalize or add indexes to reduce query time.

  4. Evaluate: When would you choose vertical over horizontal scaling?
    Sample answer

    Early stage with low traffic; when stateful constraints make horizontal complex; for short-term spikes where increasing instance size is cheapest.

  5. Explain: What’s an SLO vs. an SLA?
    Sample answer

    SLO is an internal objective (e.g., p95 < 300 ms); SLA is an external commitment with consequences; SLIs are the metrics used to judge SLO/SLA (e.g., request success rate).

  6. Apply: Users complain of intermittent slowness. What metric would you check first?
    Sample answer

    Tail latency (p95/p99) and error rate by dependency; intermittent issues often hide in tails or a single dependent service.
